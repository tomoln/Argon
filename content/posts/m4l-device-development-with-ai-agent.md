## 背景

m4l-mcp-server — M4L（Max for Live）デバイス開発のためのMCP Server — を構築し、AIエージェント（opencode）にTempoSync MIDI Generatorの生成を任せた。前回の記事「[M4Lデバイス開発をAIに委ねる: MCP Server構築記](/argon/posts/m4l-mcp-server-adventures/)」では、TypeScript + ZodでMCPツールを実装し、53テスト全PASSするところまでを扱った。

しかし、**本当の戦いはここからだった。**

MCP Server上で.maxpat（JSON形式のパッチャファイル）を生成するまでは順調だった。問題は、その.maxpatを **単一ファイルで動作する自己完結型.amxd（Ableton Liveが読み込むデバイスファイル）** にパッケージングする工程である。.amxdは単なる.maxpatのラッパーではなく、独自のバイナリチャンク構造を持ち、jsスクリプトやその他リソースを内包するためのコンテナフォーマットでもある。

本記事では、このパッケージング工程で遭遇した **5回の試行錯誤** と、そこから得た **5つの教訓** を記録する。

## 試行錯誤の概要

まず、前提条件を整理しておく。

| 項目 | 内容 |
|------|------|
| **ゴール** | Ableton Live 12 Suite で動作するテンポ同期MIDIノート生成デバイス |
| **アプローチ** | m4l-mcp-server (TypeScript製MCP Server) + opencode AIエージェントの連携 |
| **テンプレート** | 既存の`tempo-sync.json`（.maxpatのベースJSON） |
| **問題** | .maxpatから.amxdへのパッケージング工程での連続失敗 |
| **検証手段** | Ableton Liveへのドロップ → エラーメッセージ確認 → デバッグのサイクル |

テンプレートはすでに存在していた。AIが.maxpat JSONを生成するロジックもMCPツールとして実装済みだった。あとは、生成したJSONを.amxdバイナリに変換してAbleton Liveで読み込ませるだけ — そう考えていた時期が俺にもありました。

## 発生した問題（5回の試行錯誤）

結果から言えば、**5回連続で失敗した。** 1回目から5回目まで、すべて異なる原因だった。それぞれ見ていこう。

### 試行1: mx@cヘッダー問題

**症状:** 「デバイスファイルが破損しています」エラー

Ableton Liveにドロップした瞬間、以下のダイアログが表示された：

> デバイスファイル "TempoSync.amxd" が破損しています。

**原因:** mx@cヘッダーに `mxVal` フィールドが欠落していた。

.amxdの内部構造は `ampf` → `meta` → `ptch` の順にチャンクが並び、`ptch` チャンク内に `mx@c` ヘッダー + JSONペイロードが格納される。この `mx@c` ヘッダーの構造は以下の通り：

| オフセット | サイズ | 内容 | エンディアン |
|-----------|--------|------|------------|
| 0 | 4B | マジックバイト `mx@c` | ASCII |
| 4 | 4B | BE32(16) — ヘッダーサイズ | **Big Endian** |
| 8 | 4B | BE32(0) — mxVal（フラグ） | **Big Endian** |
| 12 | 4B | BE32(offset) — JSONへのオフセット | **Big Endian** |

当初、この `mxVal`（8バイト目の4バイト）を省略して12バイトのヘッダーにしていた。しかし、Ableton Live/Maxはこの値をチェックしており、`BE32(0)` を期待している。

**解決策:** `mxVal = BE32(0)` を追加し、合計16バイトのヘッダーに修正した。

このときのリファレンスは **js2max (npm)** のソースコード。このパッケージはNode.jsから.maxpat/.amxdを生成するツールであり、バイナリフォーマットの正しい実装が含まれている。

```typescript
// js2max の mx@c ヘッダー生成コードを参考に修正
function createMxcHeader(jsonOffset: number): Buffer {
  const header = Buffer.alloc(16);
  header.write("mx@c", 0, 4, "ascii");          // マジック
  header.writeUInt32BE(16, 4);                    // ヘッダーサイズ
  header.writeUInt32BE(0, 8);                     // mxVal（フラグ）
  header.writeUInt32BE(jsonOffset, 12);           // JSONオフセット
  return header;
}
```

**教訓1:** バイナリフォーマットの解析は「あるべき構造」を仮定して実装するのではなく、**リファレンス実装との比較** で検証すべき。特にエンディアンの混在（外側はLE、内側mx@cはBE）は見落としやすい。

### 試行2: js埋め込み問題

**症状:** デバイスはロードされるが何も動作しない。live.menuに "one/two/three" が表示

試行1を乗り越え、デバイス自体はロードされるようになった。しかし、MIDIノートが一切出力されない。代わりに、live.menuオブジェクトに "one/two/three" という文字列が表示されているだけで、肝心のJSコードが動作していなかった。

**原因:** Max 8の `js` オブジェクトは `embed: 1` + `text_editor_contents` を完全に無視する。

.maxpat内のjsオブジェクトには以下のようなフィールドを設定していた：

```json
{
  "box": {
    "id": "obj-1",
    "maxclass": "js",
    "text": "tempo-sync.js",
    "embed": 1,
    "text_editor_contents": "// JavaScriptコード...\nvar d = new Dict(\"tempo\");\n..."
  }
}
```

しかし、これは **Max 9のv8オブジェクト用の仕様** であり、Max 8系のjsオブジェクト（Ableton Live 12にバンドルされているMaxは8系）では機能しない。Max 8のjsオブジェクトは **外部.jsファイルからの読み込みのみ** をサポートしており、JSON内にJSコードを埋め込む方法が存在しない。

この事実を突き止めるのに時間がかかった。なぜなら、Maxのドキュメントには「embedはMax 8でもサポートされている」と書かれているからだ。しかし、実際のランタイムは違った。

**解決策:** jsオブジェクトによるスクリプティングを諦め、**ネイティブMaxオブジェクトのみで構成する** 方向に切り替えた。具体的には：

- `js` → `noteout`, `transport`, `live.dial` などのネイティブオブジェクトで代替
- テンポ同期ロジックは `metro` + `transport` の組み合わせで実現

**教訓2:** ランタイムの非ドキュメント挙動は **Web検索やリファレンス実装比較でしか判明しない**。公式ドキュメントを盲信してはいけない。特に「embed対応」のように書かれていても実際はMax 9限定というケースがある。

### 試行3: dlst埋め込み問題

**症状:** JSを.amxd内のdlstに埋め込んでもjsが読み込まない

試行2で「jsオブジェクトは外部ファイル参照が必要」とわかったため、今度は **.amxd内部にdlst（ディレクトリリスティング）エントリを作成し、そこに.jsファイルを埋め込む** というアプローチを試した。

.amxdの`ptch`チャンク内には、JSONペイロードに続いて `\n\0` + `dlst` セクションを配置できる：

```
ptch内:
  mx@c(16B, BE) + JSON(UTF-8) + \n\0 + dlst(TLV-BE)
```

`dlst` はTLV形式のディレクトリリスティングで、その中に `dire` エントリを並べる：

| フィールド | サイズ | 内容 |
|-----------|--------|------|
| type | 4B | BE32(1) または BE32(2) — ファイル種別 |
| name | 可変 (4B alignment) | ファイル名（C文字列、\0終端） |
| offset | 4B | dlst先頭からのオフセット |
| size | 4B | ファイルサイズ |

そこで、dlstに `tempo-sync.js` を登録し、ファイル本体をdlst直後に配置した。しかし、結果は同じ — jsオブジェクトがスクリプトを読み込まない。

**原因:** Max 8の `js` オブジェクトは **dlstからのファイル解決をサポートしていない。**

dlstはMaxが.amxd内にリソースをパッケージするための標準的な方法だが、それを「読む側」の実装が対応しているとは限らない。Max 8のjsオブジェクトはファイル読み込みにOSのファイルシステムAPIを使用しており、.amxd内の仮想ファイルシステム（dlst）を透過的に参照する機能は持っていない。

**教訓3:** フォーマットの機能（dlst）と、それを利用するオブジェクト（js）の対応範囲は独立している。dlstの存在を知っていても、 **読み取り側の実装がそれに対応しているとは限らない。** 事前の動作確認が不可欠。

### 試行4: パッチャJSON不完全問題

**症状:** devicewidthは効くがUIパラメータが一切表示されない

試行3でJSは諦め、ネイティブオブジェクトのみで構成した.maxpatを.amxdにパッケージングしたところ、デバイスはロードされ、`devicewidth` によるサイズ指定は反映された。しかし、肝心のUIパラメータ（live.dialやlive.numbox）が一切表示されない。

**原因:** js2maxの出力と比較して **24もの必須フィールドが欠落していた。**

当初、生成する.maxpat JSONは以下のフィールドしか持っていなかった：

```
appversion(string), classnamespace, rect, bgcolor, border,
filedrops, textdeflength, boxes, lines
```

これに対し、js2maxが出力する正しい.maxpatには **35フィールド** が含まれていた。不足していたのは以下のカテゴリ：

| カテゴリ | 不足フィールド例 |
|---------|----------------|
| バージョン情報 | `fileversion`, `appversion`（文字列→dict形式に変更） |
| デバイス定義 | `dependency_cache`, `saved_environment`, `dsp_sps_local` |
| エフェクト設定 | `canvas_selection`, `selectedlines`, `selection` |
| パラメータ設定 | `parameter_initial`, `parameter_mappable` |
| UI定義 | `varname`, `style`, `presentation_rect`, `patching_rect` |

特に深刻だったのは `appversion` のデータ型の違い：

```json
// 誤: 文字列
"appversion": "8.5"

// 正: dict形式（複数のバージョン情報を含む）
"appversion": {
  "major": 8,
  "minor": 5,
  "revision": 5,
  "architecture": "x64",
  "modern": 1
}
```

**解決策:** js2maxの出力JSONをリファレンスとして、すべてのフィールドを1つずつ比較しながら追加した。ファイル全体の差分を取ることで、見落としを防止した。

```bash
# リファレンス実装との差分確認
diff <(jq --sort-keys . my_temposync.maxpat) <(jq --sort-keys . js2max_output.maxpat)
```

**教訓4:** リファレンス実装との比較検証が最も有効なデバッグ手段。特にJSONのような構造化データは、**diffツールを使えば一発で不足フィールドが判明する。** 暗闇で手探りするより、まずリファレンスを探すべき。

### 試行5: パラメータ属性問題

**症状:** 35フィールドを追加してもUIパラメータが非表示のまま

試行4で35フィールドを追加したが、まだUIパラメータが表示されない。デバイスの背景は表示されるが、live.dialやlive.numbox、live.toggleがまるで存在しないかのように透明だった。

**原因:** UIオブジェクトに以下の2つの属性が不足していた：

```json
{
  "box": {
    "maxclass": "live.dial",
    "parameter_mappable": 1,
    "parameter_visibility": "automatable"
  }
}
```

- `parameter_mappable: 1` — このパラメータがMIDIマッピング可能であることを示す
- `parameter_visibility: "automatable"` — Liveのトラックビューでパラメータが表示されるようにする

これらはjs2maxでも自動設定されるフィールドだが、テンプレートJSONには含まれていなかった。試行4で「全フィールドをコピーした」つもりでも、box要素の内部属性までは比較しきれていなかったのだ。

**現状:** 修正済み。これによりUIパラメータが表示されるようになった（Ableton Live上での動作確認はこれから）。

**教訓5:** フィールドの有無だけでなく、**各フィールドの属性値まで検証範囲を広げる必要がある。** box内部の属性は階層が深いため、トップレベルのフィールド比較だけでは見落としが発生する。

## 発見した技術的知見

### .amxd バイナリフォーマット

一連の試行錯誤を通じて解読した.amxdフォーマットの構造をまとめる：

```
┌─────────────────────────────────┐
│ ampf チャンク (12B)            │ ← Little Endian
│   ampf(4B) + BE32(size)(4B)    │
│   + BE32(type)(4B)             │
├─────────────────────────────────┤
│ meta チャンク (12B)            │ ← Little Endian
│   meta(4B) + BE32(size)(4B)    │
│   + JSON(UTF-8, sizeバイト)    │
├─────────────────────────────────┤
│ ptch チャンク                  │ ← Little Endian
│   ptch(4B) + BE32(total_size)  │
│   ├─ mx@c ヘッダー (16B, BE)  │ ← **Big Endian**
│   │   mx@c(4B) + BE32(16)     │
│   │   + BE32(0) + BE32(offset)│
│   ├─ JSON ペイロード (UTF-8)   │
│   ├─ \n\0 (2B)                │
│   └─ dlst (TLV-BE)            │ ← **Big Endian**
│       ├─ dire エントリ1        │
│       │   type(4B, BE)        │
│       │   name(可変, \0終端)  │
│       │   offset(4B, BE)      │
│       │   size(4B, BE)        │
│       ├─ dire エントリ2...     │
│       └─ ファイル実データ      │
└─────────────────────────────────┘
```

**重要な注意点:**

1. **エンディアンの混在:** 外側のチャンク（`ampf`, `meta`, `ptch`）は **Little Endian**、内側の `mx@c` および `dlst` は **Big Endian**。この切り替わりを見落とすとオフセット計算がすべて狂う。
2. **dlstのalignment:** `dire` エントリ内のファイル名は4バイト境界にアラインする必要がある。`\0` パディングで調整する。
3. **JSONは改行\0で終端:** 多くの実装では `\n\0` をJSON終端のマーカーとして使用する。これがないとdlstのパース位置がずれる。

### Max 8 の制約まとめ

今回の試行錯誤で明らかになったMax 8の制約：

| 機能 | Max 8 | Max 9 |
|------|-------|-------|
| `js` オブジェクトの `embed: 1` | ❌ 非対応 | ✅ 対応 |
| `js` オブジェクトの `text_editor_contents` | ❌ 非対応 | ✅ 対応（v8オブジェクト） |
| `js` オブジェクトのdlst読み取り | ❌ 非対応 | ❌ 非対応（未確認） |
| `v8` オブジェクト | ❌ 非対応 | ✅ 対応 |
| ネイティブMaxオブジェクト（`noteout`等） | ✅ 最安定 | ✅ 最安定 |

**結論:** 現時点でM4LデバイスをAIエージェントに生成させるなら、「ネイティブMaxオブジェクトのみで構成する」という制約を受け入れるのが最も現実的な選択肢である。

## 学び・補足

### 5つの教訓

| # | 教訓 | 具体的内容 |
|---|------|-----------|
| 1 | **リファレンス実装を比較せよ** | バイナリフォーマットの解析は、理論より実装とのdiffが正義 |
| 2 | **公式ドキュメントを盲信するな** | ランタイムの非ドキュメント挙動はWeb検索と実験で確認する |
| 3 | **フォーマットと実装の対応は独立している** | dlstがあっても読む側が対応しているとは限らない |
| 4 | **diffツールは最強のデバッガ** | JSONのような構造化データは機械的に比較できる |
| 5 | **深い階層の属性まで検証範囲を広げよ** | トップレベルのフィールド比較だけでは不足フィールドを見逃す |

### AIエージェントによるバイナリフォーマット解析の有効性

今回の試行錯誤を通じて感じたのは、AIエージェントによるバイナリフォーマット解析は **一定の有効性があるが、限界もある** ということだ。

**有効だったこと:**
- バイナリ構造のパターン認識（エンディアンの仮説、チャンク構造の推測）
- リファレンス実装（js2max）のコード読解と比較
- 試行錯誤のログからのパターン抽出

**限界だったこと:**
- 非ドキュメント挙動の特定（embedの非対応、dlstの非対応）
- 実行時のランタイムエラーの解釈（抽象的なエラーメッセージからの原因特定）
- Max内部とのバージョン依存関係の理解（Max 8 vs 9の違い）

これらの限界は、**Web検索＋リファレンス実装比較＋実行時フィードバック** の3層検証で補完する必要がある。

### 今後の展望

今回の経験を踏まえ、M4Lデバイス開発の理想的なパイプラインは以下のように進化すると考えている：

1. **静的検証（TypeScript + Zod）**: MCPツールレベルでの型チェック
2. **フォーマット検証（バイナリ比較）**: リファレンス実装とのdiff
3. **実行時検証（Ableton Live実機）**: 最終確認

この3層検証をループエンジニアリングで高速に回すことで、AIエージェントによるM4Lデバイス開発の精度を段階的に高めていける。

現時点での実用的な制約は「ネイティブMaxオブジェクトのみ」だが、将来的にMax 9ベースのAbleton Liveがリリースされれば、v8オブジェクトによる高度なスクリプティングと組み合わせたデバイス生成も可能になるだろう。

## まとめ

m4l-mcp-server でAIエージェントにM4Lデバイスを生成させる試みは、**5回の試行錯誤** を経て少しずつ前に進んでいる。

- mx@cヘッダーのmxVal欠落 → BE32(0)追加
- js埋め込み非対応 → ネイティブオブジェクトに代替
- dlst非対応 → 同上
- 35フィールド中の24不足 → diffで全フィールド追加
- パラメータ属性不足 → parameter_mappable + parameter_visibility追加

これらの失敗は、バイナリフォーマットの深い理解とリファレンス実装の重要性を痛感させてくれるものだった。

「AIに任せる」とは、AIが全部やってくれるという意味ではなく、**AIと人間がそれぞれの得意分野を補完し合いながら、ループを高速に回す** という意味だと、改めて実感した次第である。
